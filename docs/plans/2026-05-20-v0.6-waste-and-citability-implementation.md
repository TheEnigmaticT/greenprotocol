# GreenChemistry.ai v0.6 Waste + Citability Implementation Plan

> **For Hermes:** Use subagent-driven-development only after this plan is accepted. Keep scope limited to waste scoring, recommendation metadata, citability, and drill-down UX. Do not blur this work into broader 0.6 route-redesign tasks.

**Goal:** Implement a defensible v0.6 waste-scoring layer and per-recommendation release stamping so GC.ai can show one simple waste score, retain deeper waste calculations for ranking and auditability, and make every recommendation citable by exact software version.

**Architecture:** Extend the Python chemistry service to compute and return a structured `wasteAnalysis` object, then persist it inside `analysis_result` in the Next.js app. Extend recommendation metadata so each recommendation has a single primary waste-facing benefit plus citation metadata derived from a single version source. Update results UI to show a cockpit-style top-line waste summary with optional drill-downs.

**Tech Stack:** Next.js 16 App Router, TypeScript, Python/FastAPI chemistry service, Supabase JSONB persistence, PubChem/GHS/CHEM21 evidence, package.json version source.

---

## Scope guardrails

This plan covers:
- waste score architecture
- waste storage shape

- version stamping / citability
- UI drill-down behavior

This plan does **not** cover:
- full upstream LCA ingestion
- full SDS crawler buildout
- methodology paper writing
- broader route-redesign generation

## Task 1: Establish version source of truth and analysis metadata

**Objective:** Make the app expose one canonical GC.ai version string and persist it with each saved analysis.

**Files:**
- Modify: `package.json`
- Create: `lib/version.ts`
- Modify: `lib/types.ts`
- Modify: `app/api/analyze/route.ts`
- Modify: `app/api/analyses/[id]/route.ts`
- Modify: `app/api/analyses/route.ts`

**Step 1: Keep `package.json` version authoritative**

Confirm the app version is sourced from:

```json
{
  "version": "0.1.0"
}
```

Do not hardcode the release in UI components or prompt templates.

**Step 2: Create a version helper**

Create `lib/version.ts` that exports:
- `GCAI_VERSION`
- `getAnalysisMetadata()` helper returning timestamp + version

Suggested shape:

```ts
import pkg from '@/package.json'

export const GCAI_VERSION = pkg.version

export function getAnalysisMetadata() {
  return {
    generatedAt: new Date().toISOString(),
    gcaiVersion: GCAI_VERSION,
    methodologyVersion: 'waste-v0',
  }
}
```

If importing `package.json` directly becomes awkward in the current TS config, use a tiny generated module or env bridge. Do **not** duplicate the version manually.

**Step 3: Extend analysis types**

In `lib/types.ts`, add:
- `AnalysisMetadata`
- `WasteAnalysis`
- `RecommendationCitationMetadata`

Attach them to:
- `AnalysisResult`
- `Recommendation`
- `AnalysisSummary` shape if needed for list pages

**Step 4: Persist metadata on create**

In `app/api/analyze/route.ts`, before insert:
- stamp `analysisResult.analysisMetadata`
- ensure every recommendation carries citation metadata or inherits analysis-level metadata

**Step 5: Preserve metadata on patch**

In `app/api/analyses/[id]/route.ts`, PATCH should not silently drop `analysisMetadata` or per-recommendation citation metadata when accepted/rejected recommendations are saved.

**Step 6: Verify list/detail routes return metadata**

`app/api/analyses/[id]/route.ts` and `app/api/analyses/route.ts` should continue returning the full `analysis_result` JSON with the new metadata intact.

**Step 7: Verification**

Run:

```bash
npm run lint
```

Then inspect a saved analysis payload and confirm:
- `analysis.analysisMetadata.gcaiVersion` exists
- `analysis.analysisMetadata.generatedAt` exists

**Step 8: Commit**

```bash
git add package.json lib/version.ts lib/types.ts app/api/analyze/route.ts app/api/analyses/[id]/route.ts app/api/analyses/route.ts
git commit -m "feat: add canonical version metadata to saved analyses"
```

## Task 2: Define chemistry-service waste analysis models

**Objective:** Add a stable structured waste-analysis response model in the Python service.

**Files:**
- Modify: `services/chemistry/scoring/models.py`
- Modify: `services/chemistry/models.py`
- Modify: `services/chemistry/main.py`
- Create: `services/chemistry/scoring/waste_analysis.py`
- Create: `services/chemistry/scoring/waste_helpers.py`

**Step 1: Add response models in Python**

In `services/chemistry/scoring/models.py` or a nearby shared scoring model file, add typed structures for:
- waste summary
- direct waste
- hazard segments
- liquid burden
- process burden
- upstream placeholders
- evidence sources

Keep these separate from `PrincipleScore` so the waste model remains reusable even if principle scoring changes.

**Step 2: Add app-facing transport models**

In `services/chemistry/models.py`, extend HTTP response models so `/score` can return a top-level `waste_analysis` object in addition to principle scores.

**Step 3: Create waste helper utilities**

In `services/chemistry/scoring/waste_helpers.py`, add deterministic utilities for:
- summing solvent vs non-solvent mass
- summing liquid mass in kg
- bucketing H-codes into hazard groups
- counting hazard-flagged chemicals
- safe handling of missing `quantity_kg`

**Step 4: Create waste scoring module**

In `services/chemistry/scoring/waste_analysis.py`, compute:
- direct waste totals
- toxic / CMR / flammable / corrosive / environmental buckets
- liquid handled/discarded kg
- process burden passthrough or derived fields
- top-line waste impact score
- primary driver sentence
- confidence

Keep the formula explicit and comment the weight choices.

**Step 5: Wire the module into `/score`**

In `services/chemistry/main.py`:
- call the new waste module
- include `waste_analysis` in the response
- do not remove existing principle scoring

**Step 6: Verification**

From repo root or service dir, run the relevant test or curl/manual verification path you already use for the chemistry service. At minimum verify the JSON now includes:

```json
{
  "waste_analysis": {
    "summary": {
      "wasteImpactScore": 0,
      "grade": "A"
    }
  }
}
```

**Step 7: Commit**


**Files:**

**Step 1: Update principle prompt instructions**

In `lib/prompts/principles.ts`, instruct the model that each recommendation must surface one main workflow-relevant reason such as:
- reduces toxic waste
- cuts liquid cleanup burden
- reduces purification burden
- lowers direct chemical waste

**Step 2: Update schema expectations if needed**

If the current tool schema in `lib/pipeline.ts` is too thin, extend it to request:
- `primaryBenefit`
- `secondaryBenefits`

Keep the schema strict enough that the UI can trust it.

**Step 3: Add fallback derivation**

If the model omits `primaryBenefit`, derive it in `lib/pipeline.ts` from known principle numbers or waste deltas so the UI never lacks a lead explanation.

**Step 4: Keep evidence expandable**

Do not inflate the visible recommendation copy with every metric. Preserve detailed evidence in `evidence`, `wasteDelta`, and drill-down fields.

**Step 5: Verification**

Run a sample analysis and confirm recommendation cards can display a short label like:
- “Primary benefit: reduces toxic waste”

without requiring the evidence drawer to be open.

**Step 6: Commit**


# GreenChemistry.ai v0.6: Waste Scoring and Citability Architecture

## 1. Goal

v0.6 should make waste a first-class product surface instead of a side effect of carbon scoring.

The system needs to do two things at once:
1. give the scientist **one simple thing to know** on the report card and on each recommendation
2. preserve a **deep, auditable waste model** underneath for ranking, drill-down, export, and citation

At the same time, every recommendation should become citable as software output by carrying the exact GreenChemistry.ai release that generated it.

---

## 2. Product Principle

### Default experience: cockpit, not dashboard sprawl

The default UI should behave like a 747 cockpit:
- one top-line waste score
- one primary explanation
- one best next action

Everything else is optional drill-down.

### Internal model: deep, structured, reusable

The system should still compute and store deeper waste calculations so they can:
- improve recommendation ranking
- support evidence-backed explanations
- power exports and academic citation
- support future upstream/LCA work without redesigning the data model again

---

## 3. Why waste must be separate from carbon

Waste and carbon are related but not interchangeable.

If waste is collapsed into CO2e, GC.ai becomes easier to message to climate buyers but less credible to chemists. The waste model should therefore stay distinct from emissions.

### Product rule
- **Waste** is its own score family
- **Carbon/emissions** remain adjacent but separate
- recommendation copy may mention carbon when helpful, but carbon does not replace waste in the core explanation

---

## 4. v0.6 waste model

The waste model should be MECE enough to support product decisions without overwhelming the user.

## 4.1 Top-line metric

### Waste Impact Score
A single composite score or grade shown on the report card.

This score should summarize:
- direct chemical waste mass
- toxic / hazard-weighted waste burden
- liquid handling / liquid discard burden
- process burden tied to cleanup, transfers, vessels, and purification

It should not include upstream/LCA burden by default unless the underlying evidence is available and confidence-labeled.

---

## 4.2 Stored sub-metrics

These should be stored even when not shown by default.

### A. Direct chemical waste
What mass of chemicals becomes waste in the process.

Core fields:
- `total_chemical_waste_kg`
- `product_mass_kg`
- `waste_to_product_ratio`
- `solvent_waste_kg`
- `non_solvent_waste_kg`

### B. Toxic / hazard-segmented waste
What fraction of waste carries meaningful health, environmental, or operational burden.

Core fields:
- `toxic_waste_kg`
- `cmr_waste_kg`
- `corrosive_waste_kg`
- `flammable_waste_kg`
- `environmentally_hazardous_waste_kg`
- `hazard_flagged_chemicals[]`

This is where GHS-backed hazard classification matters.

### C. Liquid burden
The product should normalize liquid burden in **kilograms**, not litres, so solvent density differences are respected.

Core fields:
- `liquid_handled_kg`
- `liquid_discarded_kg`
- `liquid_recovered_kg` (optional, future)
- `solvent_mass_by_step[]`

### D. Process burden
Waste is not only a materials problem. It is also a workflow problem.

Core fields:
- `vessel_count`
- `transfer_count`
- `solution_prep_count`
- `wash_count`
- `extraction_count`
- `purification_count`
- `recrystallization_count`
- `process_burden_score`

This is the broader, more product-legible framing for what was loosely called “glassware economy.”

### E. Upstream / LCA placeholders
Not required for core v0.6 scoring, but the data model should reserve space for it.

Core fields:
- `upstream_included`
- `upstream_method`
- `upstream_confidence`
- `embodied_waste_kg`
- `embodied_co2e_kg`
- `upstream_notes`

This allows future LCA work without pretending it is ready for deterministic scoring now.

---

## 5. Suggested composite score behavior

The exact weights can change, but the structure should be stable.

### Proposed score inputs
- direct waste mass score
- hazard-weighted waste score
- liquid burden score
- process burden score
- optional upstream modifier
- confidence modifier

### Product rule
The composite score should be explainable in one sentence.

Example:
> Waste Impact: C+ — driven mainly by solvent-heavy cleanup and one high-concern reproductive-toxicity solvent.

The user should not need to inspect a long table to understand why the score landed where it did.

---

## 6. Recommendation behavior

Recommendations should consume the deep waste model but not expose all of it by default.

## 6.1 One primary reason per recommendation

Every recommendation should carry a single workflow-relevant reason selected from the deeper model, for example:
- reduces toxic waste
- cuts liquid cleanup burden
- reduces purification burden
- lowers direct chemical waste
- lowers upstream burden

A recommendation may improve many things, but the UI should lead with one.

## 6.2 Secondary reasons

Stored but optional:
- score deltas
- hazard flags removed
- SDS-backed disposal/handling notes
- evidence citations
- supporting sub-metrics

---

## 7. SDS role

SDS matters, but it should not be the primary scoring substrate.

### Use SDS for
- disposal/handling context
- supplier-specific warnings
- mixture / formulation nuance when relevant
- recommendation trust and evidence
- scientist drill-down and export support

### Do not use SDS as the main score engine because
- vendor formats vary
- supplier language is inconsistent
- mixtures complicate normalization
- it is harder to produce deterministic, comparable scoring from raw SDS prose

### Product rule
Use structured hazard sources for scoring.
Use SDS as an evidence and workflow-context layer.

In practice that means:
- PubChem/GHS and curated sources drive the actual score
- SDS links/notes strengthen the explanation and safety context

---

## 8. Data model additions

## 8.1 Analysis-level metadata

Each saved analysis should preserve a stable software identity.

Suggested fields:
- `analysisMetadata.generatedAt`
- `analysisMetadata.gcaiVersion`
- `analysisMetadata.methodologyVersion` (optional but recommended)
- `analysisMetadata.pipelineModel` (optional)
- `analysisMetadata.chemistryServiceVersion` (optional, future)

### Product rule
The release number shown to the user must come from a single source of truth.
Current best candidate: `package.json` version, surfaced through a small app helper rather than duplicated manually.

## 8.2 Analysis-level waste object

Suggested shape:

```json
{
  "wasteAnalysis": {
    "summary": {
      "wasteImpactScore": 82,
      "grade": "B+",
      "primaryDriver": "Solvent-heavy cleanup and toxic solvent burden",
      "confidence": "calculated"
    },
    "directWaste": {
      "totalChemicalWasteKg": 12.4,
      "productMassKg": 0.9,
      "solventWasteKg": 9.8,
      "nonSolventWasteKg": 2.6
    },

      "toxicWasteKg": 3.1,

```

## 8.3 Recommendation-level metadata

Each recommendation should carry enough information to be citable and explainable.

Suggested fields:
- `primaryBenefit`
- `secondaryBenefits[]`
- `wasteDelta` object
- `citationMetadata.gcaiVersion`
- `citationMetadata.generatedAt`
- `citationMetadata.analysisId` (once persisted)
- `evidence.sdsReferences[]` (optional)

## 9. UX model

## 9.1 Report card default view
Show:
- Waste Impact score/grade
- one sentence on the main driver
- one best next action

## 9.2 Expandable waste details
Optional panels/tabs:
- Waste breakdown
- Hazard breakdown
- Liquid burden
- Process burden
- Evidence / SDS / citations

## 9.3 Recommendation cards
Default card content:
- original chemical/process issue
- recommended alternative or redesign
- one primary reason
- visible software version / citation affordance

Expandable card content:
- evidence
- hazard details
- SDS notes
- deeper waste deltas
- literature citations

---

## 10. Citability requirements

Scientists already cite tools like DOZN. GC.ai should therefore act like a versioned scientific instrument.

### Minimum requirements
- stable product URL
- stable release/version identifier
- visible or inspectable version on every recommendation/result
- preserved version in saved analyses

### Recommended next-step requirements
- `Cite this analysis` text block or export field
- methodology document or preprint later
- stable methodology version if scoring logic changes materially between releases

### Product rule
A recommendation should answer:
> Which GC.ai release generated this advice?

without requiring engineering access or source inspection.

---

## 11. Rollout recommendation

### v0.6 core
Ship:
- composite waste score
- deep structured waste record
- kg-based liquid burden
- hazard-segmented waste
- process burden integration
- version stamping on every recommendation and saved analysis
- optional drill-down UI

### v0.6 later / optional
Add:
- SDS caching / richer SDS evidence extraction
- upstream/LCA augmentation
- export-ready citation blocks
- methodology versioning separate from app version if needed

---

## 12. Non-goals for this slice

This spec does **not** require:
- full cradle-to-gate LCA scoring
- vendor-agnostic SDS parsing perfection
- external publication infrastructure yet
- a final academic citation format

It does require that the architecture stop blocking those things.

---

## 13. Definition of success

v0.6 succeeds here if GC.ai can:
1. present one clean waste score without flattening everything into carbon
2. store deeper waste calculations in a reusable structure
3. rank and explain recommendations from those calculations
4. let scientists inspect detail only when they want it
5. show exactly which software release generated each recommendation
